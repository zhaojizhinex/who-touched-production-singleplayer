# Agent 对话交互记录

## 文档说明

本文档不是原始逐字聊天导出，而是基于当前仓库代码、已有归档和交付结果整理出的制作过程对话记录。目标是还原项目从想法、收敛、实现到打包交付的 Agent 协作轨迹，便于比赛展示、复盘和后续维护。

## 项目基础信息

- 项目名：`谁动了生产环境`
- 当前形态：单机版软件项目题材推理游戏
- 当前仓库目录：
  - `app/`：可运行项目源码
  - `docs/`：设计与归档文档
- 当前技术方案：
  - 前端：React + TypeScript + Vite
  - 桌面封装：Electron
  - 构建与打包：PowerShell 脚本

## 交互记录整理

### 阶段一：项目方向确认

#### 用户目标

用户提出要做一个“谁动了生产环境”主题的项目，希望把程序员、测试、项目经理、实习生等角色放进一个带有事故、甩锅、排查、投票的推理场景里，做成可以展示的作品。

#### Agent 关键判断

- 直接做多人联机版本，工程量和不确定性都偏高，不适合作为短周期可交付方案。
- 题材最重要的不是复杂联网，而是“软件交付现场的混乱感”和“谁在搞事”的推理氛围。
- 最现实的首发版本应该先收敛成单机版，让玩家与 AI 队友完成一整局可玩的流程。

#### 交互结论

- 先做单机可玩版，而不是先做联机框架。
- 保留“阵营对抗 + 事故修复 + 会议投票”三条主干。
- 用软件项目交付场景替代传统狼人杀或太空狼人杀语境。

### 阶段二：玩法收敛

#### 初始讨论焦点

围绕“玩法要不要复杂”“要不要保留太多 mini game”“角色要不要太多”“房间是不是太分散”进行了收敛。

#### Agent 建议

- 保留回合制，而不是做实时移动，减少操作复杂度。
- 保留房间概念，但房间数量控制在玩家能快速理解的范围。
- 每回合给玩家一次核心决策机会，让节奏更紧凑。
- AI 需要能自动行动、制造事故、修事故、开会和投票，这样单机版才能成立。

#### 收敛结果

当前落地版本采用以下规则：

- 总人数 5 人：1 名玩家 + 4 名 AI
- 角色池 5 个：
  - 开发
  - 后端
  - 测试
  - 软件经理
  - 实习生
- 房间 5 个：
  - 工位区
  - 会议室
  - 测试区
  - 发布区
  - 茶水间
- 正常阵营每轮 1 个操作节点
- 搞事阵营每轮 2 个操作节点
- 会议、投票、胜负结算全部保留

### 阶段三：核心系统设计

#### 对话重点

为了让单机版不是“点一下就结束”，项目进入了规则和状态机设计阶段。

#### Agent 设计思路

- 用统一 `GameState` 管理整局状态。
- 把角色、房间、事故、线索、会议、日志、投票拆成独立数据结构。
- 用一个集中式回合结算函数推进整局。
- 玩家动作和 AI 动作都走同一套主循环，减少逻辑分叉。

#### 当前代码落点

- 状态定义：`app/src/types.ts`
- 游戏主逻辑：`app/src/game.ts`
- 页面交互：`app/src/App.tsx`

#### 对话产出结论

- 项目最终采用“动作选择 -> 回合结算 -> 事件推进 -> 会议/投票 -> 下一轮”的单局循环。
- 不使用复杂后端存档，不接数据库，所有局内数据内存中运行。

### 阶段四：角色专属交互加入

#### 用户诉求

仅靠“做事 / 排查 / 修问题”几个大按钮，角色区分度不够，展示感也偏弱。

#### Agent 调整方案

将原本抽象的一次动作，改造成“动作类型 + 角色专属交互”的两层结构：

- 做事或修问题时，必须从 3 个专属互动选项中选 1 个
- 每个选项带有：
  - 风险等级
  - 成功率
  - 成功分值
  - 失败分值

#### 具体实现结果

每个角色都拥有 3 个对应语义明显的交互项，例如：

- 开发：
  - 代码重构
  - 代码检查
  - 代码优化
- 后端：
  - 链路重整
  - 接口排查
  - 参数兜底
- 测试：
  - 全量回归
  - 重点复测
  - 抽样检查

#### 设计收益

- 玩家每轮都有一次明确的风险收益选择。
- 角色之间的差异不只体现在数值上，也体现在行为语义上。
- 展示时更容易讲清楚“这个职业为什么会这样做”。

### 阶段五：AI 行为与会议发言

#### 问题

单机版最容易失真的是 AI。如果 AI 只是随机数驱动，会让推理部分很空。

#### Agent 处理方向

- 区分普通阵营 AI 与搞事阵营 AI 的决策逻辑。
- 普通阵营优先修严重事故、排查高价值线索、偶尔用技能。
- 搞事阵营会主动制造事故、混入修复、带会议节奏、转移怀疑。
- 给每个 AI 角色加人格标签和不同发言语气。

#### 当前实现结果

AI 具备以下行为能力：

- 推任务
- 排查
- 修问题
- 用技能
- 制造事故
- 拉会
- 根据怀疑值投票

会议发言会结合以下因素生成：

- 角色人格
- 当前活跃事故
- 最近线索
- 场上最高怀疑目标

### 阶段六：界面和可读性优化

#### 交互问题

规则成型后，界面一度存在三个问题：

- 信息很多，但玩家第一眼抓不到重点
- 人物卡和发言内容不够好区分
- 新玩家上手成本仍偏高

#### Agent 采取的改动

- 增加首页说明和开局规则提示
- 增加当前目标提示文案
- 用颜色区分不同角色卡片
- 增加地图式房间展示和房间热度信息
- 保留日志区、线索区、会议摘要区，形成清晰的信息流

#### 结果

当前版本已经具备完整的比赛展示 UI：

- 首页
- 规则引导
- 主局内界面
- 会议投票界面
- 结算界面

## 制作过程中的关键决策摘要

### 为什么放弃多人联机首发

- 联机、房间同步、断线重连和服务端裁决都不适合当前交付节奏。
- 比赛展示更需要“能跑、能玩、能解释”，而不是架构过重。
- 单机版已经足够体现题材、规则和设计能力。

### 为什么把人数压到 5 人

- 5 人局更适合单机 AI 推理场景。
- 玩家更容易跟踪每个人的位置、发言和嫌疑变化。
- 界面信息密度更可控。

### 为什么保留房间和事故

- 房间让行动有空间语义。
- 事故让回合推进有公共压力。
- 会议和投票才不会变成纯随机猜人。

### 为什么要引入角色专属交互

- 让“职业”不只是名字。
- 增加风险收益决策。
- 提高展示时的可讲述性。
